作成日: 2026-06-15（v0.2でAWSベースの技術構成を反映）
更新日: 2026-06-17（v0.3でバックエンドホスト構成・HULFT Square書き戻し方式を確定）
ステータス: ドラフト（合意済みの設計判断を反映。未確定事項は末尾参照）
インフラ方針: 基本的にAWS上に構築する

## 1. サービス概要

営業担当者がAI顧客エージェントを相手に、音声でヒアリング（ディスカバリー商談）のロールプレイを行うトレーニングサービス。

最大の特徴は、**同一の見込み顧客に対して複数回の商談を重ね、顧客の「真の課題」を段階的に掘り下げていく**点にある。各回の終了後に先輩・上司から評価を受け、全回終了後に総評を受け取ることで、単発の練習ではなく一連の営業プロセスとして学習できる。

### コンセプトの核

- 同一顧客との連続商談を通じた「課題の掘り下げ」体験
- AI顧客は回をまたいで一貫した人物であり続け、良いヒアリングに応じて課題への気づきが進行する
- 顧客の「真の課題」は全回終了までユーザーに非公開とし、見抜けたかどうかを最後に答え合わせする

---

## 2. 登場人物・ロール

| ロール | 説明 |
|---|---|
| 営業ユーザー | ロープレを実施する本人。プログラム作成・各回の設定・音声ヒアリングを行う |
| 評価者（先輩／上司） | プログラムに事前設定される。各回の評価および全回終了後の総評を記入する |
| ロープレ用Slack Bot | ロープレ完了イベントを受けて、先輩側のチャンネル／DMへ通知を送信する |

---

## 3. 用語定義

- **プログラム**: 1つの見込み顧客との商談全体。複数のヒアリングセッションで構成される最上位の単位
- **ヒアリングセッション**: 制限時間内に行う1回の音声やり取り。本サービスの最小単位
- **真の課題**: AI顧客が内部に持つ、隠された本質的な課題。全回終了までユーザー非公開
- **気づき度**: AI顧客が自身の真の課題にどれだけ気づいているかの内部状態。回を追って変化する

---

## 4. 全体フロー

1. ユーザーがプログラムを作成（分野・総ヒアリング回数を選択）
2. システムがLLMでAI顧客の「真の課題」と性格設定を自動生成し、プログラムに固定
3. ユーザーが1回目の目標と制限時間を設定し、ヒアリング開始
4. AI顧客エージェントと音声（プッシュトゥトーク）でやり取り
5. 制限時間到達でセッション終了 → システムが会話を自動判定し、気づき度・状態を更新
6. ロープレ完了イベント発火 → HULFT Square経由でSlack Botが先輩へ評価依頼を通知（評価ページのリンク付き）
7. 先輩が評価ページで当該回の評価を記入
8. ユーザーは履歴画面で評価を閲覧可能（読むことは必須でなく、次回開始とは独立）
9. ユーザーが次回の目標・制限時間を設定し、ヒアリング開始（3〜8を繰り返す）
10. 設定回数すべて完了 → 総評依頼イベント発火 → Slackで先輩へ通知
11. 先輩が全回を通じた総評を記入
12. 総評ページで、総評と「真の課題」をユーザーに開示

## 5. 機能要件

### 5.1 プログラム管理
- 分野の選択（金融、流通 など）
- 総ヒアリング回数の選択
- プログラム作成時にLLMでAI顧客の真の課題・性格設定を自動生成し固定（プロンプトで軽く制約をかける）
- 日をまたいでもセッション単位で中断・再開できる（進行中のプログラム状態を永続化）

### 5.2 AI顧客エージェント
- プログラム作成時に生成・固定される「正データ」と、回ごとに更新される「状態」を分けて管理（詳細は §6）
- 性格タイプ（例: 課題に気づいている／まだ気づいていない 等）を保持
- 各セッションのシステムプロンプトは「正データ＋直近までの状態＋その回の振る舞いルール」で構成
- セッション終了時に、会話内容をLLMで判定し、気づき度・ラポール度・開示済み情報・各回サマリを更新（**システムの自動判定**で進行。先輩評価には影響されない）

### 5.3 ヒアリングセッション
- 各回ごとに「その回の目標」と「制限時間（分）」をユーザーが入力
- 音声によるやり取り（プッシュトゥトーク方式。将来変更の可能性あり）
- 制限時間が近づいても会話が締まりそうにない場合は、終盤でAIが締めへ誘導する。ユーザーが既に締めに向かっている場合は打ち切らず、ユーザーが終えるまで待つ
- 会話の途中中断は不可（原子的）。異常終了（切断・離脱等）した場合はそのセッションを無効破棄してやり直し
- 会話の文字起こしを保存。あわせてセッション全体の音声を通しで録音し、常時保存する（先輩が最初から最後まで再生できる）
- セッションのタイトルは、会話内容をもとにLLM（Bedrock）で自動生成する

### 5.4 評価機能（先輩）
- 先輩用の評価入力ページ（③）に以下を表示: その回の目標／分野／何回目か／**真の課題**／ロープレの録音音声（最初から最後まで通しで再生可能）／整形済みの文字起こしテキスト
- 同ページで当該回の評価を記入できる
- **複数の先輩がそれぞれ評価を記入する**（1セッションに複数の評価が紐づく）
- 全回終了後、総評入力ページで総評を記入できる

### 5.5 ユーザー向け表示
- **評価一覧ページ（①）**: これまでのロープレを、タイムスタンプ・タイトルとともに一覧表示
- **評価詳細ページ（②）**: 選択したロープレについて、先輩からの評価（複数）の詳細を閲覧
- 真の課題は全回終了まで非表示
- 全回終了後、総評ページで総評と真の課題を開示

## 6. データモデル（概要）

### Program（プログラム）
- id / ユーザーID / 分野 / 総ヒアリング回数 / 状態 / 作成日時
- 紐づく評価者（先輩）

### CustomerProfile（顧客・正データ：生成後固定）
- program_id / 業界 / 企業規模 / 相手の役職 / 表層ニーズ / **真の課題** / 性格タイプ / 初期の気づき度

### CustomerState（顧客・状態：回ごとに更新）
- program_id / 現在の気づき度 / ラポール度 / 開示済み情報リスト / 各回サマリ

### HearingSession（ヒアリングセッション）
- id（UUID） / program_id / 回数（n回目） / 目標 / 制限時間（分） / タイトル（LLM自動生成） / 状態 / 開始日時 / 終了日時 / 文字起こし / 録音（通し・常時保存）

### Evaluation（評価）
- id / session_id / 評価者ID / 内容 / 作成日時 （複数の先輩がそれぞれ記入。1セッションに複数の評価が紐づく）

### OverallReview（総評）
- id / program_id / 評価者ID / 内容 / 作成日時

### プログラム
作成済み → 進行中 → 全回完了 → 総評依頼済み → 総評完了（クローズ）

未開始 → 進行中 → ┬→ 完了 → 評価依頼送信済み →（任意で）評価済み
　　　　　　　　　 └→ 異常終了 → 破棄（同一回をやり直し）

## 9. システムアーキテクチャ

### 9.0 全体方針（開発演習向け）
本サービスは本番運用ではなく開発演習として作成し、数回の実証実行に耐える構成とする。フロントエンド（frontend/）は当面ローカル（localhost）で動作させ、常駐バックエンド（backend/）は **Amazon ECS Fargate 上で常時ホスト** する。バックエンドの前段には **Application Load Balancer（ALB）** を配置し、HTTPS および WebSocket（WSS）のエンドポイントを提供する。AWS は音声認識・音声合成・LLM・ストレージ・コンピュートのマネージドサービスとして利用する。本番向けのサーバーレス基盤（Lambda / EventBridge / Step Functions / CloudFront 等）は今回は組まず、将来の本番化時の選択肢として残す。リージョンは東京（ap-northeast-1）を想定。

#### 責務分担
- **AWS** が担う処理: ①音声→テキスト（STT）／②テキストから回答生成（LLM）／③回答の読み上げ（TTS）／④データ保存・状態管理／⑤バックエンドの常時ホスト（ECS Fargate + ALB）。
- **HULFT Square** が担う処理: ①ヒアリング終了イベントを受けて、指定された先輩のSlack DMへ評価リンクを送信／②ロープレで発生したJSON等のテキストを、評価ページに貼れるテキスト形式（md・txt等）へ整形し、**バックエンドAPIへ書き戻す**。

### 9.1 コンポーネント対応表

| 機能 | 採用構成 |
|---|---|
| フロントエンド | React + React Router のSPA（frontend/）。ローカル（localhost）で実行。AWSホスト（S3+CloudFront/Amplify）は当面対象外 |
| バックエンド | **FastAPI**（backend/）を **Amazon ECS Fargate** 上で常時稼働。AWS SDK（boto3）で Transcribe / Bedrock / Polly / S3 / DB を利用し、ストリーミングや WebSocket も扱う。ASGI サーバーは Uvicorn |
| ロードバランサー | **Application Load Balancer（ALB）**。HTTPS 終端（ACM 証明書）、HTTP API および WebSocket（WSS）の振り分け、ヘルスチェック、固定エンドポイントの提供。ロープレ中の長時間 WebSocket 接続を考慮し、アイドルタイムアウトは 300 秒以上に設定 |
| コンテナレジストリ | Amazon ECR（Fargate タスクの Docker イメージ格納） |
| 認証（ユーザー・先輩） | 演習スコープでは簡易認証で可（先輩用ページはUUID URLで保護するためログイン不要）。本番化時は Amazon Cognito |
| HULFT Square 連携認証 | バックエンドの内部 API へは共有 API キー（`X-API-Key` ヘッダ等）で認証。キーは AWS Secrets Manager で管理し、HULFT Square 側にのみ配布（iPaaS に AWS IAM キーは渡さない） |
| データストア（Program / Session / Evaluation / OverallReview / CustomerProfile / CustomerState） | PostgreSQL。開発はローカル（Docker等）、実証時は小規模な Amazon RDS PostgreSQL でも可 |
| 音声・文字起こし・録音の保存 | Amazon S3 |
| 音声認識（STT・日本語） | Amazon Transcribe（ストリーミング） |
| 音声合成（TTS・AI顧客の声） | Amazon Polly（日本語ニューラルボイス） |
| LLM | Claude（Amazon Bedrock経由）。会話応答は高速・低コストモデル、課題生成・終了時判定/タイトル生成は高性能モデルに使い分け |
| 外部連携（HULFT Square） | バックエンド → HULFT Square: HTTP でトリガー。HULFT Square → バックエンド: 整形済みテキストの **API キー認証付き内部 API** へ書き戻し |
| シークレット管理 | ローカル開発は環境変数／AWSプロファイル。AWS 上は Secrets Manager / SSM Parameter Store。ECS タスクロールで AWS サービスへアクセス（アクセスキーの埋め込みは行わない） |
| 監視・ログ | 演習スコープでは最小限（アプリログ + CloudWatch Logs） |

### 9.2 ヒアリング1ターンのデータフロー（プッシュトゥトーク）
1. クライアント（React）がPTTで発話を録音し、音声をバックエンドへストリーミング送信（WebSocket等）
2. バックエンドが Amazon Transcribe（ストリーミング）で文字起こし
3. Claude（Bedrock・高速モデル）で顧客応答を生成（CustomerProfile＋CustomerState＋その回の目標をコンテキストに付与）
4. Amazon Polly で音声合成し、クライアントで再生
- 上記を制限時間に達するまで繰り返す
- セッション全体の音声は通しで録音し S3 に保存する（先輩が最初から最後まで再生できるようにするため）

### 9.3 セッション終了時の処理（バックエンドで実行）
1. 制限時間が近づいたら終盤で締めへ誘導（ユーザーが締めに向かっていれば終えるまで待つ）し、セッション終了
2. 通しの録音音声と全文の文字起こしを確定し、S3 に保存。あわせてセッションの生データを PostgreSQL に永続化
3. Claude（Bedrock・高性能モデル）が会話を判定し、気づき度・ラポール度・開示済み情報・各回サマリを更新（CustomerState を更新）。あわせて会話内容からセッションタイトルを自動生成
4. ロープレで発生したデータ（文字起こし・目標・分野・回数・真の課題などのJSON等）を組み立て、評価ページ URL（UUID）を含めて HULFT Square へ渡すペイロードを準備
5. バックエンド（ECS Fargate）が HULFT Square を HTTP でトリガー
6. HULFT Square が、(a) 上記 JSON を評価ページ用の md/txt へ整形し、**ALB 経由のバックエンド内部 API**（API キー認証）へ POST して DB を更新（評価ページが自動で読み込む）、(b) 指定された先輩の Slack DM へ評価リンクを送信
7. 全回完了時は同様にバックエンドが HULFT Square をトリガーし、総評依頼を通知（総評用の書き戻しも同様に内部 API 経由）

### 9.4 Slack 連携
- HULFT Square 経由で、指定された先輩（複数可）のSlack DMへ通知（送信実体は連携・Bot）
- 各回完了時: 評価依頼通知（**先輩用の評価入力ページ③**のリンク）／全回完了時: 総評依頼通知（総評入力ページのリンク）
- 送信先は先輩個人のDMであり、ユーザーは投稿先に不可視。これにより真の課題の非公開性を担保
- 評価ページのURLには推測不可能なUUIDを用いる（認証基盤は作らず、URLの推測・総当たりによる閲覧を防ぐ）

### 9.5 HULFT Square（iPaaS）
バックエンドとSlack／評価ページの間に立つ連携・整形層として配置し、以下2つを担う。

1. **通知**: バックエンドからのトリガーを受けて、指定された先輩のSlack DMへ評価リンク（全回完了時は総評リンク）を送信する。
2. **テキスト整形**: ロープレで発生したJSON等のデータを、評価ページ表示用のテキスト形式（md・txt等）へ変換する。

#### 書き戻し方式（確定）: バックエンド API キー認証経由

HULFT Square から S3 へ直接書き込む方式は採用しない。整形後の md/txt は **バックエンドの内部 API へ POST** し、FastAPI が PostgreSQL を更新する。評価ページはバックエンド API 経由で表示データを取得する（先輩による手動貼り付けは行わない）。

**採用理由（要約）**
- データの正（PostgreSQL）をバックエンドに一元化できる
- HULFT Square（iPaaS）に AWS IAM キーを渡す必要がなく、漏洩リスクを下げられる
- 評価ページ・一覧・詳細が同一の API 経路でデータを参照できる
- 冪等な再送・ログ記録をバックエンド側で制御しやすい

**連携フロー**
1. バックエンド → HULFT Square: HTTP トリガー（セッション終了／全回完了イベント）
2. HULFT Square: JSON を md/txt へ整形
3. HULFT Square → バックエンド: `POST /internal/evaluation-artifacts`（仮）等の内部 API へ、`X-API-Key` ヘッダ付きで整形済みテキストを送信
4. バックエンド: API キーを検証し、該当セッション（またはプログラム）の表示用フィールドを DB 更新
5. HULFT Square: 先輩の Slack DM へ評価ページ URL を送信

**運用上の注意**
- API キーは AWS Secrets Manager で管理し、HULFT Square の接続設定にのみ設定する
- 同一 `session_id`（または `program_id`）への再 POST は上書きでよい（冪等）
- 内部 API はインターネット到達可能（ALB 経由）だが、API キー未所持のリクエストは拒否する

### 9.6 デプロイ構成（バックエンド）

```
[フロントエンド localhost] ──HTTPS/WSS──▶ [ALB] ──▶ [ECS Fargate / FastAPI]
[HULFT Square] ──HTTPS（内部API・APIキー）──▶ [ALB] ──▶ [ECS Fargate / FastAPI]
[ECS Fargate] ──▶ RDS PostgreSQL / S3 / Bedrock / Transcribe / Polly / HULFT Square
```

| 項目 | 構成 |
|---|---|
| コンピュート | ECS Fargate（常時 1 タスク以上） |
| ロードバランサー | ALB（HTTPS + WSS） |
| イメージ | Docker → ECR |
| デプロイ | GitHub Actions 等で ECR プッシュ → ECS サービス更新（詳細は実装時に定義） |

## 10. 非機能要件（暫定）

- 本要件は開発演習スコープを前提とし、可用性・冗長化・監視は最小限とする（数回の実証実行に耐えればよい。本番化時に強化）
- 音声認識は日本語の精度を重視（Amazon Transcribe ストリーミング）
- レイテンシはリアルタイム必須ではないが、ターン制の会話として違和感のない範囲
- セッション全体の音声を通しで録音・保存し、先輩が最初から最後まで再生できる
- 異常終了時はセッションを安全に破棄し、データ不整合を残さない
- 評価ページ等のURLにはUUIDを用い、推測・総当たりによる閲覧を防ぐ
- AWSの認証情報・Slackトークン等の機密は環境変数等で安全に管理し、リポジトリに含めない
- （本番化時）IAM最小権限・保存時暗号化・VPC・CloudWatch監視・Bedrock等のコスト監視を整備

### 構成 — 確定済み
- フロント: React + React Router のSPA、ローカル実行（AWSホストは対象外）
- バックエンド: **FastAPI** を **ECS Fargate** 上で常時ホスト（Lambda/EventBridge/Step Functions等のサーバーレス基盤は不採用）
- ロードバランサー: **ALB**（HTTPS 終端、WebSocket 対応、固定エンドポイント）
- HULFT Square 書き戻し: **バックエンド内部 API + API キー認証**（S3 直書きは不採用）
- データストア: PostgreSQL（開発はローカル、実証は小規模RDSも可）
- 音声認識: Amazon Transcribe ストリーミング
- 音声: 通しで録音・保存し、先輩が全編再生可能
- Bedrock: 高速モデル（会話）／高性能モデル（生成・終了時判定・タイトル生成）の使い分け
- ※具体的なClaudeモデルIDは、実装時にBedrockの現行ラインナップを確認して選定
