# 商談参考資料の添付機能 実装説明

## 変更内容・実装内容の概要
新規商談作成時に、営業資料や製品情報などの参考資料（PDF、TXT、MDファイル）を添付し、AI顧客がその前提情報を踏まえてロールプレイ応答を行うことができる機能を追加しました。
また、UI上の「新規プログラム作成」という表記を、より分かりやすい「新規商談作成」へ統一しました。

---

## 修正詳細

### 1. データベース・スキーマ定義の調整
- **`Program` モデルとレスポンススキーマ**:
  - `materials_text` (データベース上のText型。抽出したテキストを格納) および `materials_filename` (ファイル名) のマッピングを、`to_response` 関数で `ProgramResponse` に正しく連動するように反映しました。

### 2. バックエンドAPI (`POST /api/programs/{program_id}/upload-material`)
- **ファイルアップロード・パース処理**:
  - アップロードされたファイルをメモリ上で受け取り、サイズが10MBを超えていないかを検証します（DoS対策）。
  - 拡張子が `.pdf`, `.txt`, `.md` のいずれかであることを厳密にチェックします。
  - PDFファイルは `pypdf` ライブラリを使用してテキストを抽出し、テキスト/マークダウンファイルは UTF-8 デコードを行います。
  - 抽出したデータをプログラムモデルに保存し、DB更新を行います。

### 3. LLM プロンプトへの参考資料インジェクション
- **`audio_pipeline.py` & `hearing.py`**:
  - `build_system_prompt` に引数 `materials_text` を追加。
  - ロールプレイ時に WebSocket の接続情報から `materials_text` を取得し、LLMに渡す `system_prompt` へ `{materials_section}` として動的に挿入します。

### 4. フロントエンドUIおよびAPI接続
- **`api.ts` & `types.ts`**:
  - `uploadProgramMaterial(programId: string, file: File)` クライアント関数を実装し、`FormData` を介してバックエンドにバイナリデータをポストします。
- **`SettingsPage.tsx` & `HomePage.tsx`**:
  - 画面全体の表記を「プログラム作成」から「商談作成」へと置換・修正しました。
  - ページ右カラムに「3. 参考資料の添付 (任意)」セクションを追加し、ファイル入力要素およびファイル選択時のサイズ制限バリデーション（最大10MB）を組み込みました。
  - 商談作成ボタンがクリックされた際、プログラムが正常に作成された直後に、ファイルが選択されていれば `uploadProgramMaterial` を連続して実行し、成功後に次のセットアップ画面へ遷移するようにしました。

## 【追記：2026/06/23】AI顧客が添付資料を認識しない問題の対策

添付された参考資料がAI顧客に正しく参照されない・認識されていないという事象に対して、以下の2点の強化・修正を行いました。

### 1. システムプロンプトにおける解釈指示の追加 (`audio_pipeline.py`)
AI顧客（LLM）のプロンプト内に「参考資料について」の明示的なガイダンスを追加しました。
- 添付された参考資料（`materials_text`）が「営業担当から事前に送付された資料」であることを教示。
- 営業担当から「送った資料」「添付ファイル」「事前テキスト」などの言及があった場合は、この資料を指していると理解し、その内容に沿って応答するように振る舞いを修正。

### 2. 空ファイル・テキスト抽出不可時のバリデーション強化 (`program_service.py`)
ファイルアップロードは成功したものの、中身が空であったりPDF等からテキストが一切抽出できなかった場合に、そのまま登録されてAI顧客に空のデータが渡るのを防ぐバリデーションを追加しました。
- PDFからテキストが1文字も抽出できなかった場合（スキャン画像のみのPDF等）、エラー（`ValueError`）を発生させてアップロードを却下するように改善。
- テキスト・マークダウンファイルが空文字の場合も同様に却下。

### 3. マイグレーション時の重複カラムエラー回避 (`004_program_materials.py`)
- すでにデータベース側に手動等で `materials_text` 等のカラムが追加されている環境において、コンテナ起動時の `alembic upgrade head` がエラー終了してサーバー自体が起動しなくなる（結果として API 全体が `Not Found` または `Failed to fetch` になる）のを防ぐため、`try-except` ブロックを用いて安全にマイグレーション（カラム追加）を実行する設計に変更し、起動時の耐障害性を最大化しました。

## 具体的な利用例（シナリオ例）

### 状況
あなたは「A社向けのセキュリティソリューション」の提案商談ロールプレイを行いたいとします。
手元には、今回の商談で提案する予定の製品仕様書 `security_solution_spec.pdf` があります。

### 操作手順
1. **商談の作成**:
   - トップ画面の「▶ 新規商談作成」ボタンを押します。
   - 業界やIT知識レベルを入力します。
   - 「3. 参考資料の添付 (任意)」のファイル選択欄で、`security_solution_spec.pdf` を選択します。
   - 「▶ 商談作成」ボタンを押します。
2. **ロールプレイの開始**:
   - 商談セッションが開始され、AI顧客との音声対話が始まります。
3. **対話中の挙動**:
   - あなたが「弊社の製品には、不正アクセスを検知した際に管理者に自動アラートを送信する機能が含まれています」と発言します。
   - AI顧客は、添付された `security_solution_spec.pdf` の内容（例：「アラート送信機能は標準でSlackおよびメール連携に対応」）を裏で参照します。
   - AI顧客は「仕様書に書いてあるSlack連携のアラート機能について詳しく教えてほしい」といった、添付資料の具体的な仕様に紐づいた質問やリアクションを返してきます。
